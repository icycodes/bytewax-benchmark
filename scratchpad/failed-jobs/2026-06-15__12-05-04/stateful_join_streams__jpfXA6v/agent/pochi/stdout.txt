## How it works

### Pipeline stages

```
FileSource(impressions.jsonl)          FileSource(clicks.jsonl)
        │                                       │
  op.key_on  (extract user_id)           op.key_on  (extract user_id)
        │                                       │
  op.map  (parse JSON line → dict)       op.map  (parse JSON line → dict)
        │                                       │
        └──────────── op.join ──────────────────┘
                 (stateful inner join)
                         │
                     op.map
              (format → JSON string)
                         │
               FileSink(joined.jsonl)
```

### Key design decisions

| Choice | Reason |
|---|---|
| `emit_mode="complete"` | Inner-join semantics — a record is only emitted when **both** an impression **and** a click exist for the same `user_id`. Users seen in only one stream are silently dropped. |
| `insert_mode="last"` | Keeps the most-recent event per side per key, avoiding unbounded state growth if duplicates arrive. |
| Two-stage keying (`key_on` → `map`) | `key_on` produces `(key, raw_line)` so parsing is deferred until after keying, keeping the key extraction lambda lightweight. |
| Formatted stream stays keyed `(user_id, json_str)` | `FileSink` is a `FixedPartitionedSink` — its internal `part_fn` requires the upstream item to be a `(key, value)` tuple for routing. |
