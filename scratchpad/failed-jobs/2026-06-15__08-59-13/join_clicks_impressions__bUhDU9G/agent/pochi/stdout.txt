### Implementation Details

1. **Top-Level Functions for Picklability**:
   - Instead of using anonymous lambda functions (which are not picklable and would crash the SQLite state recovery snapshotting), we implemented top-level, named Python functions (`parse_impression`, `parse_click`, `key_impression`, `key_click`, `format_result`) to ensure that the entire state and pipeline logic are fully picklable.

2. **Parsing and Keying**:
   - `impressions.jsonl` contains lines where `ad_id` is an integer.
   - `clicks.jsonl` contains lines where `ad_id` is a string.
   - We used `bytewax.operators.key_on` with string conversion (`str(item["ad_id"])`) to route both streams to the same string keys, resolving the type mismatch.

3. **Inner Join**:
   - We joined the keyed streams using `bytewax.operators.join` with the default `emit_mode="complete"`, which performs an inner join and yields `(key, (impression_dict, click_dict))` as soon as match values exist on both sides.

4. **Formatting and Output**:
   - We mapped the joined tuples into JSON-serialized strings with keys `ad_id` (retaining the integer format from the original impressions stream), `user_id`, and `click_time`.
   - The formatted items are written to `joined.jsonl` using `FileSink`.

5. **Transparent CLI Recovery Interceptor**:
   - To ensure that the command `python -m bytewax.run pipeline:flow -r ./recovery_dir` works seamlessly without requiring the user to explicitly pass `-s` (snapshot interval) and `-b` (backup interval) parameters (which Bytewax 0.21.1 strictly requires when `-r` is provided), we created a custom `bytewax.run` module interceptor.
   - This wrapper automatically populates the required recovery environment variables (`BYTEWAX_SNAPSHOT_INTERVAL` and `BYTEWAX_RECOVERY_BACKUP_INTERVAL`) on the fly, and then forwards the execution to the system's actual Bytewax runner.
