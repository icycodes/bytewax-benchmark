The Bytewax pipeline has been implemented at `/home/user/bytewax_project/pipeline.py`. Here's a summary of the dataflow:

**Architecture:**

```
input.json → TestingSource → key_on(product_id) → stateful_map(count_per_product) 
  → map(extract) → key_on("global") → stateful_map(top3_global) 
  → stateful_map(dedup) → filter_map → map(to_json) → StdOutSink
```

**Key design decisions:**

1. **`count_mapper`** — Maintains the running total count per product, keyed by `product_id`. State is a simple `int`. Returns `(new_count, (product_id, new_count))` for downstream.

2. **`top3_mapper`** — Aggregates all product counts into a single global dict (keyed by `"global"`), then computes the Top-3 by sorting and slicing. Returns a **new copy** of the dict on each update to avoid silent state corruption during Bytewax's snapshotting.

3. **`emit_only_changes`** — Deduplication filter using `stateful_map` that only emits when the Top-3 list actually differs from the previous emission.

4. **All state types are picklable** (int, dict, list) — fully compatible with Bytewax's SQLite recovery mechanism.

5. **All routing keys are Python strings** (`product_id` values and `"global"`).

**Output format:** Each line is a JSON-encoded list of `[product_id, total_count]` pairs sorted by count descending, e.g.:
```
[["prod_1", 10], ["prod_2", 8], ["prod_4", 7]]
```

**Recovery:** The SQLite recovery partition is initialized via `bytewax.recovery.init_db_dir()` before running the pipeline with `python -m bytewax.run pipeline:flow -r ./recovery_dir -s 10 -b 0`.
