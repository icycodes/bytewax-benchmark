**Pipeline architecture:**

1. **Input** — Reads `impressions.jsonl` and `clicks.jsonl` via `FileSource`
2. **Parse** — Maps each JSON line string through `json.loads` to get dictionaries
3. **Key** — Uses `op.key_on` to extract `user_id` as the state key for both streams
4. **Join** — Uses `op.join` with default `emit_mode="complete"` (inner join semantics — emits only when both an impression and click exist for the same `user_id`)
5. **Format** — Uses `op.map_value` to transform the joined `(impression, click)` tuple into a JSON string containing `user_id`, `impression`, and `click` fields
6. **Output** — Writes to `joined.jsonl` via `FileSink`

**Key design decisions:**
- `op.map_value` (not `op.map`) is used after the join to preserve the `(key, value)` tuple structure required by Bytewax's routing layer
- Each line in `joined.jsonl` is a JSON object with `user_id`, `impression`, and `click` fields
- Users with only an impression or only a click are correctly excluded (inner join behavior)
